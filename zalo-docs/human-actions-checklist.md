# ARIA — Complete Human Actions Checklist

Everything you must do manually that the code cannot do for you.
Work through each section top to bottom. Sections marked **[CRITICAL]** block
the app from working if skipped.

## 0 — Security: rotate compromised CRON_SECRET [CRITICAL]

The CRON_SECRET value `Jz3Jy2DvLWE3VBNtrPMa8UGALnkWf27KkpcZu1yT9gg=` was
committed to git history and must be treated as compromised.

> **Why `ALTER DATABASE SET` doesn't work:** Supabase blocks that command for
> the `postgres` role. Use **Supabase Vault** instead — it's Supabase's official
> encrypted secret store and works fine from the SQL editor.

**Generate a new secret:**
```bash
openssl rand -base64 32
```

Copy the output. Use it in steps below.

**0a — Store secrets in Supabase Vault (SQL editor — not in git)**

Supabase dashboard → **SQL Editor** → run these two statements:
```sql
SELECT vault.create_secret(
  'https://aria-consult.vercel.app',
  'aria_base_url',
  'ARIA app URL for cron jobs'
);

SELECT vault.create_secret(
  'YOUR-NEW-CRON-SECRET',
  'aria_cron_secret',
  'ARIA cron authentication secret'
);
```
Replace `YOUR-NEW-CRON-SECRET` with the value you generated above.

These are stored **encrypted** in Supabase Vault and are **never in git**.
The pg_cron jobs read them at runtime via `vault.decrypted_secrets`.

To rotate the secret later (no migration needed):
```sql
UPDATE vault.secrets SET secret = 'NEW-SECRET' WHERE name = 'aria_cron_secret';
```

**0b — Apply the remediation migration**
This rewrites all four cron jobs to read from vault instead of the old
hardcoded values. Run AFTER step 0a (vault must exist first):
```bash
npx supabase db push
```
Migration file: `supabase/migrations/20260702000000_fix_pg_cron_use_settings.sql`

**0c — Update Vercel with the new CRON_SECRET**
Vercel → your project → **Settings → Environment Variables** → `CRON_SECRET`
→ Edit → paste the NEW value (same one you put in vault) → Save.

**0d — Redeploy Vercel**
Vercel → **Deployments** → **Redeploy**. This picks up the new `CRON_SECRET`.

## 1 — Supabase: enable required extensions [CRITICAL]

These extensions must be enabled before any migrations run. Do this ONCE per
Supabase project.

Supabase dashboard → **Database → Extensions**, enable:

| Extension | Purpose |
|-----------|---------|
| `pg_cron` | Scheduled cron jobs inside Postgres |
| `pg_net` | HTTP calls from cron jobs to your API |

After enabling, run your migrations:
```bash
npx supabase db push
```

## 2 — Supabase: store secrets in Vault [CRITICAL]

Must be done BEFORE running the pg_cron migrations (vault secrets must exist
before the cron jobs try to read them). If you already did step 0a, skip this.

> **Note:** `ALTER DATABASE SET` is blocked by Supabase. Use Vault instead.

In **Supabase SQL Editor**:
```sql
SELECT vault.create_secret(
  'https://aria-consult.vercel.app',
  'aria_base_url',
  'ARIA app URL for cron jobs'
);

SELECT vault.create_secret(
  'YOUR-CRON-SECRET',
  'aria_cron_secret',
  'ARIA cron authentication secret'
);
```

To verify secrets exist:
```sql
SELECT name, description FROM vault.decrypted_secrets
WHERE name IN ('aria_base_url', 'aria_cron_secret');
```

To rotate the CRON_SECRET later (no new migration needed):
```sql
UPDATE vault.secrets SET secret = 'NEW-SECRET' WHERE name = 'aria_cron_secret';
```
Then update `CRON_SECRET` in Vercel to match.

## 3 — Vercel: set all environment variables [CRITICAL]

Vercel → your project → **Settings → Environment Variables**.
Add each variable for **Production** environment (and Preview if you want it to
work in preview deploys too).

### Core (always required)

| Variable | Where to get it |
|----------|----------------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase → Project Settings → API → Project URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase → Project Settings → API → anon/public key |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase → Project Settings → API → service_role key |
| `ANTHROPIC_API_KEY` | console.anthropic.com → API Keys |
| `CRON_SECRET` | Generated by you (`openssl rand -base64 32`) |

### Email (required for email notifications)

| Variable | Value |
|----------|-------|
| `RESEND_API_KEY` | Resend dashboard → API Keys |
| `EMAIL_FROM` | e.g. `ARIA <no-reply@aria.vn>` (must match verified domain) |

### Zalo (add when you register OA — can be done later)

| Variable | Where to get it |
|----------|----------------|
| `ZALO_APP_ID` | Zalo for Developers → your app → App ID |
| `ZALO_SECRET_KEY` | Zalo for Developers → your app → Secret Key |

> Without `ZALO_APP_ID` / `ZALO_SECRET_KEY`, all Zalo features gracefully
> degrade (return "skipped"). Everything else still works.

After adding/changing any variable: Vercel → **Deployments** → **Redeploy**.

## 4 — Anthropic: get an API key [CRITICAL]

1. Go to [console.anthropic.com](https://console.anthropic.com).
2. Sign up or log in.
3. **API Keys** → **Create Key**.
4. Copy and add to Vercel as `ANTHROPIC_API_KEY`.
5. Add a credit card under **Billing** and add $10–$20 in credits to start.

> ARIA uses `claude-sonnet-4-6` by default. Monitor spend in the
> [Anthropic console](https://console.anthropic.com) under **Usage**.

## 5 — Resend: email domain verification [CRITICAL for email]

1. Sign up at [resend.com](https://resend.com).
2. **Domains** → **Add Domain** → enter your domain.
3. Add the DNS records Resend shows you (SPF, DKIM, DMARC).
4. Wait for DNS propagation (5–30 min), click **Verify**.
5. Create an API key (**API Keys** → **Create API Key**, Sending access only).
6. Add to Vercel: `RESEND_API_KEY` and `EMAIL_FROM`.

See [email-setup.md](./email-setup.md) for full instructions.

---

## 6 — Custom domain: DNS configuration [optional but recommended]

If you want `aria-consult.vercel.app` to be `yourapp.com`:

1. Vercel → your project → **Settings → Domains** → Add domain.
2. Vercel shows you DNS records (CNAME or A record).
3. Add those records in your DNS provider.
4. Wait for propagation.
5. Update `app.base_url` in Supabase:
   ```sql
   ALTER DATABASE postgres SET app.base_url = 'https://yourapp.com';
   ```
6. Apply the remediation migration again (`npx supabase db push`).
7. Update `EMAIL_FROM` in Vercel to use your domain.

## 7 — Supabase: verify cron jobs are running

After deploying, check the cron jobs are scheduled:

In **Supabase SQL Editor**:
```sql
SELECT jobname, schedule, active FROM cron.job ORDER BY jobname;
```

Expected output:
```
generate-daily-briefing   | 0 0 * * *    | true
refresh-zalo-token        | 5 * * * *    | true
send-email-notifications  | 15 * * * *   | true
send-zalo-notifications   | 5 * * * *    | true
```

To see recent cron execution history:
```sql
SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 20;
```

## 8 — Zalo OA setup [do when ready — not blocking]

See [zalo-oa-setup.md](./zalo-oa-setup.md) for the complete step-by-step guide.

Summary of manual steps:
1. Register at [id.zalo.me](https://id.zalo.me) with a Vietnamese phone number.
2. Create an Official Account at [oa.zalo.me](https://oa.zalo.me).
3. Create an app at [developers.zalo.me](https://developers.zalo.me).
4. Link the OA to the app.
5. Submit for OA verification (takes 3–7 business days).
6. Add `ZALO_APP_ID` and `ZALO_SECRET_KEY` to Vercel.
7. Connect Zalo in the ARIA Settings page using your personal Zalo numeric user ID.

Before connecting Zalo in the settings page, you must first **follow your own
OA from your personal Zalo account** — Zalo cannot send messages to users who
have not followed the OA.

## 9 — Supabase Auth: set production URL [CRITICAL]

If you skip this, Supabase confirmation emails link back to `localhost:3000`
and the OTP token expires before you can use it.

Supabase dashboard → **Authentication → URL Configuration**:

1. **Site URL** — change from `http://localhost:3000` to `https://www.aria-consult.cloud`
2. **Redirect URLs** — add `https://www.aria-consult.cloud/**`

Save. Then resend your confirmation email (sign up again with the same address,
or use Supabase dashboard → **Authentication → Users** → find your user →
**Send confirmation email**).

## 10 — First login and account setup

1. Open your deployed ARIA URL.
2. Sign up with your email address.
3. Open **Settings** → fill in:
   - Owner name (used in greeting messages)
   - Industry type
   - Business context (a few sentences about your business — this shapes AI responses)
4. Check **Email notifications** is toggled on if you set up Resend.
5. Add your first deal in the **Deals** tab.
6. Let the daily briefing cron generate tomorrow's briefing (runs at 00:00 UTC /
   07:00 Vietnam time).
